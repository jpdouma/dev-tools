# UNIVERSAL CONTEXT HUB: OPERATIONAL MANUAL

## 1. The Core Philosophy
The Universal Context Hub is a local LLM Orchestration and State Management Server. It bridges the gap between human product management, architectural planning, and autonomous code generation.

Large Language Models (LLMs) suffer from context degradation and token truncation over long sessions. This tool solves that by enforcing a **Compartmentalized Handoff Architecture**:
1. **The Product Manager (Human):** Operates the Dev Tool UI, manages the state, and defines the active workspace.
2. **The Architect (Gemini Context Window):** A fresh web-based LLM session initialized with the exact current state of the project. Used for brainstorming, architectural mapping, and writing execution tasks.
3. **The Syntax Writer (`gemini-cli`):** The terminal-based AI that executes the strict, deterministic tasks generated by the Architect.

By freezing the state and moving from context window to context window, the AI ecosystem never loses its train of thought.

---

## 2. The Semantic Boundary: Manual vs. Blueprint

To prevent AI hallucination, the system enforces a strict semantic boundary:
* **`OPS_MANUAL.md` (This Document):** The static instructions for the Hub. It dictates *how to use the tool*. It is dynamically injected into the SudoLang prompt but kept separate from the project state.
* **`PROJECT_BLUEPRINT.md` (The Living Architecture):** The dynamic, project-specific documentation (e.g., the Yield Force 5-Zone Architecture). This is the active source of truth for the project.

## 3. UI Component Precision Map

### A. The Master Canvas (Markdown Editor)
The main window handles the `PROJECT_BLUEPRINT.md`. 
* **CRITICAL PARSER RULE:** The backend JSON converter relies on a strict Markdown heading hierarchy. 
  * `# ` (H1) is strictly reserved for the **Document Title**. Using multiple H1s will overwrite previous sections.
  * `## ` (H2) is strictly reserved for **Parts/Major Sections**.
  * `### ` (H3) is strictly reserved for **Chapters**.
  * `#### ` (H4) and below are for standard sub-sections within chapters.

### B. The Context Drawer (Right Panel)
* **Session Tab:** Manages the volatile state. The `Local Scratchpad` is private to the human. The `Changelog` and `Next Objectives` are injected directly into the LLM hand-off prompt.
* **Blueprint Tab:** A deprecated, read-only view of the raw `core_state.json` payload generated by the Markdown Editor.
* **Envs Tab:** Manages tenant-specific JSON payloads (e.g., configuring `ecu_medical` vs. `red2roast`).
* **ADRs Tab:** The immutable ledger of Architecture Decision Records. Supports bulk deletion and granular edits.
* **Prompts Tab:** The SudoLang master template editor. This dictates the system rules for the Architect.

### C. The Tooling Layer
* **+ Quick ADR Modal:** Features a "Smart Auto-Import" polyglot parser. Humans can paste either a raw JSON array or LLM-generated Regex text (Title: ... Friction: ...) and the tool will automatically chunk them into a reviewable queue.
* **🔗 Link Source Code:** Connects the active workspace to a local OS directory. Drag-and-drop a folder to auto-fill the absolute path. This enables the backend to run `git diff` and inject live codebase context into the prompt.

## 4. The SudoLang Command Reference

The Hub relies on a customized SudoLang framework to force the Architect (Gemini) to output highly deterministic, machine-readable data.

### `/ADR [topic]`
* **Purpose:** Formalizes a single architectural rule.
* **Output:** A strict raw JSON object (`title`, `scope`, `friction`, `ruling`, `boundary`).
* **Usage:** Paste into the Smart Auto-Import modal.

### `/ADR bulk`
* **Purpose:** Batch-processes multiple architectural rules simultaneously.
* **Output:** A raw JSON array of ADR objects.
* **Usage:** Paste directly into the Smart Auto-Import modal to queue them for human review and database commitment.

### `/CLI_TASKS`
* **Purpose:** Translates brainstormed architecture into atomic execution steps for the terminal CLI.
* **Output:** A raw JSON array containing `target_file`, `objective`, and `instructions`.
* **Usage:** Feed directly into `gemini-cli` in the terminal.

### `/UPDATE_BLUEPRINT`
* **Purpose:** Updates the living `PROJECT_BLUEPRINT.md` for the active workspace.
* **Output:** Raw Markdown text adhering strictly to the H1/H2/H3 hierarchy rules.
* **Usage:** The human copies the Markdown output and pastes it into the Hub's UI Markdown Editor to update the state.

### `/WRAPUP`
* **Purpose:** Executes the State Freeze before the context window degrades (approx. every 4 hours).
* **Output:** A lean JSON diff (`completed_steps`, `next_objectives`, `unresolved_issues`).
* **Usage:** Paste this diff into the UI's Session Tab to prime the `core_state.json` for the next context window.

## 5. Standard Operating Procedure (The 4-Hour Loop)

1. **Initialization:** Human selects workspace, clicks **Copy Prompt**, and pastes the payload into a fresh Gemini Web session.
2. **Brainstorming:** Architect and Human align on the technical approach based on the injected Codebase Context.
3. **Formalization:** Architect outputs `/ADR bulk` and `/CLI_TASKS`. Human executes code via terminal and saves ADRs via the UI.
4. **Documentation:** Architect runs `/UPDATE_BLUEPRINT` to ensure the project architecture is permanently documented.
5. **State Freeze:** Architect runs `/WRAPUP`. Human updates the Session Tab, closes the degraded window, and loops back to Step 1.
